# MatrixOne v23.0.7.0 Release Report

Congratulations! MatrixOne v23.0.7.0 was officially released on February 23, 2023. In this release, with the cloud-native architecture and full database feature set established, v0.7.0 focuses on targeted stability and performance improvements:

- **Stability**: overall memory-management mechanism was overhauled, significantly reducing the probability of memory leaks. Timeout handling in several modules was adjusted, so that edge-case data loading and restarts no longer cause the system to stop responding. The extreme case of a distributed-architecture-wide crash triggered by a single failed module was also fixed.
- **Performance**: this iteration includes extensive read / write optimization. On the read side: Zonemap-based statistics yield better execution plans; `CLUSTER BY` on table creation lays data out ahead of time for faster access; numerous hotspots in the `IN` operator, type conversion, and predicate filtering are optimized. On the write side: large bulk parallel writes directly from CN nodes to shared storage bring significant load-performance improvements.

```
docker pull matrixorigin/matrixone:0.7.0
```

## Latest features

- New parallel-load data mode.
- Added `Foreign Key`.
- Added `CLUSTER BY` column on `CREATE TABLE`.
- Added `MYSQL_COMPATBILITY_MODE` parameter to manage MySQL-specific compatibility behavior.
- Changed the default isolation level to `REPEATABLE_READ` for better MySQL compatibility.
- Full unique-constraint implementation for `UNIQUE INDEX`.
- `LOAD` from client-local files.
- Added `ALTER VIEW`.
- Added function to query SQL results.
- Added tenant-pause functionality.
- Multi-tenant: added cluster tables (admins can write to a specific tenant).
- Added `GROUP_CONCAT` aggregate.
- Added `FORMAT`, `REPLACE`, `CURDATE`, `FIELD`, `SUBSTRING_INDEX` and other system functions.

## Known issues

- The v0.7.0 storage format is not compatible with earlier versions — direct upgrade is not possible.
- OLTP concurrent performance is not yet stable.
- Secondary-index acceleration is not yet implemented.
- 100GB-scale TPCH benchmarks are unstable.
- Long-running, high-frequency stress tests can cause OOM.
- Loading multiple GB-scale tables in a distributed cluster can cause OOM.
