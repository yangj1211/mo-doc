# MatrixOne's Positioning

In a vast, complex data tech stack full of database products, MatrixOne is positioned as a SQL relational database focused on one-stop convergence and flexible scalability. The design goal: provide a database as simple to use as MySQL, capable of handling OLTP, OLAP, and other workloads and data types — while auto-sensing workload and data-volume changes for fast elastic scaling — simplifying today's complex "multi-database + ETL" data architectures.

The closest database to MatrixOne in the current industry is SingleStore. Both use a unified storage model, support OLTP / OLAP / a mix of other data workloads and types, and take cloud-native + flexible scalability as core architectural tenets.

![mo_vs_singlestore](https://community-shared-data-1308875761.cos.ap-beijing.myqcloud.com/artwork/docs/overview/mo-other-database/mo_vs_singlestore.png)

- **Architecture**: MatrixOne is fully cloud-native and containerized. MatrixOne borrows Snowflake's [cloud-native data warehouse](https://event.cwi.nl/lsde/papers/p215-dageville-snowflake.pdf) compute-storage separation — storage goes to shared cloud storage; the compute tier is stateless containers. To handle OLTP-style fast writes, MatrixOne adds TN + LogService — using block storage to support high-frequency writes. Raft three-replica consistency ensures WAL HA; WAL data is then asynchronously flushed to shared storage. Unlike MatrixOne, SingleStore extends from a share-nothing architecture into cloud-native compute-storage separation — only putting cold data into shared storage (see [SingleStore architecture paper](https://dl.acm.org/doi/pdf/10.1145/3514221.3526055)) — still requiring data sharding and rebalancing. MatrixOne, like Snowflake, is fully on shared storage — no sharding.

- **Workload types**: MatrixOne takes HTAP as its core, expanding into stream processing, time-series, ML, search, etc. On the HTAP front, unlike TiDB's "dual engine" path (separate TP / AP engines), MatrixOne delivers HTAP on a single storage engine and a single compute engine. The storage engine is primarily disk-backed columnar + in-memory row — similar to SingleStore. TiDB-style paths require internal dual-engine sync; TP and AP data must be stored separately. MatrixOne avoids that sync and dual storage.

- **User experience**: MatrixOne targets MySQL compatibility — wire protocol, SQL syntax, and core features. MatrixOne only adds unique syntax for things MySQL doesn't support — streaming writes, vector types, etc. For some MySQL advanced features with low usage (triggers, stored procedures), MatrixOne hasn't added support yet. Overall, MySQL-based applications port easily to MatrixOne — MySQL ecosystem tools like Navicat / DBeaver (visual admin and modeling), DataX / Kettle (ETL), Spark / Flink (compute) all work via MySQL connectors against MatrixOne.
