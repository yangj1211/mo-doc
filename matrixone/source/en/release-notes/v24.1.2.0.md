# MatrixOne v24.1.2.0 Release Report

We are thrilled to announce the official release of MatrixOne kernel v24.1.2.0 on May 20, 2024.

## What is MatrixOne?

MatrixOne is a distributed, hyper-converged, heterogeneous cloud-native database. It aims to provide a cloud-native, high-performance, highly elastic, MySQL-compatible HSTAP database — letting users complete the full data-processing pipeline for transactional, analytical, time-series, and streaming workloads in a single system.

## Highlights

This iteration significantly expands MatrixOne's capabilities. Key features include snapshot backup, incremental physical backup, CTAS, BITMAP, and vector indexes — steadily positioning MatrixOne as an ideal choice for enterprise data management and analytics.

### Snapshot backup and restore (Beta)

Snapshot backup creates read-only copies at a specific point in time, protecting data efficiently without disrupting database operation. This iteration brings tenant-level snapshot backup / restore — on data loss or corruption, snapshots restore state quickly, reducing RTO. Snapshots don't require stopping the database service, simplifying backup flows and preserving business continuity. They play a key role in disaster recovery by ensuring rapid recovery of critical data.

### Incremental physical backup (Enterprise)

On top of the existing full physical backup in the `mo_backup` tool, incremental physical backup is now available. In large-volume environments, full backups take long and consume storage. Incremental backup automatically tracks the last backup time and previously backed-up objects — only new objects need to be backed up.

### CTAS

This iteration adds CTAS (Create Table As Select). CTAS is a SQL statement that quickly creates a new table from existing data. It combines table creation and a select query, offering an efficient way to take table snapshots, transform data, or create data models for reporting and analysis. Characteristics:

- Efficient — executed as one atomic operation, cutting the separate create-table and insert steps.

- Concise — creating a new table and filling data in one shot simplifies data modeling and ETL.

- Flexible — supports most of `SELECT`'s capabilities: `WHERE`, `JOIN`, `GROUP BY`, etc., enabling complex transformations.

### BITMAP for fast distinct counts

Data analysis often requires distinct counts over large datasets. `COUNT(DISTINCT values)` can be slow at scale. This iteration introduces BITMAP functions, providing faster and cheaper dedup, count, query-stats, and sort operations. For BI, user-behavior analysis, and real-time recommendation, BITMAP significantly speeds up processing and improves accuracy.

### Vector index

The previous iteration added vector types and common similarity functions; this iteration adds vector indexing, substantially lifting performance on vector data. Vector indexes enable fast retrieval of the closest or most-similar data points to a query vector — essential for large-scale search and matching. In ML/AI — with LLMs, RAG, and similar workloads increasingly processing high-dimensional data with complex similarity and pattern matching — vector indexes reduce compute and significantly improve query latency, optimizing overall system performance.

- `CREATE INDEX ... USING IVFFLAT`
- `ALTER TABLE ... ALTER REINDEX`

- `bit` data type

### Indexes and constraints

- Vector index

### Functions and operators

- `SYSDATE` date function
- `TO_BASE64` / `FROM_BASE64` encoding
- `MD5` / `SHA1` hashes
- `SUBVECTOR` — extract a sub-vector
- `SERIAL_EXTRACT` — extract sub-elements from serial / tuple values
- `CLUSTER_CENTERS` — compute cluster centers for a vector column
- Scalar-vector arithmetic (+, −, ×, /)

### System parameters

- `keep_user_target_list_in_result`
- `foreign_key_checks`

### MySQL compatibility

- Refactored CSV reader and splitting for MySQL compatibility.

## Known Issues

- Vector indexes currently only accelerate `l2_distance`-metric queries.
- Only tenant-level snapshot backup/restore is supported; cluster-level is not yet.
- Snapshot restore requires writing all data once, with noticeable CPU/memory cost.
- Occasional OOM under large data volumes.
- Occasional system hang under high-concurrency workloads.
